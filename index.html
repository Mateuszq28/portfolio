Map Scrap Detector

Script made in "Harpagan Online" event

Matlab script, written fast during the Harpagan Online. It searches small, modificated scrap of map (dfferent resolution, noise, some new shapes on it) on the bigger one.

github startup message (pol.) tasks (pol.)

The Harpagan Online event took place on April 17, 2020. Each participant had to search for prepared small photos of map sections on a large map of the Geoportal portal. After finding a matching location, it was necessary to view it in Google Street View and note the answers to the question related to it. Finding the clippings was quite demanding, so I came up with the idea of ‚Äã‚Äãwriting a script in Matlab that would do this task for me. The cuts were made on an unknown scale. In addition, they contained distortions in the form of circles and numbers with the numbers of the point to be found. I decided to take screenshots of a map showing the area on a large scale. Then I applied the cutout to the large image and moved it in a loop over the entire large image. I also solved the differences in scale with a loop - guessing and then searching for the best matching scale progressing in a loop. To account for minor differences between the slice and the map, I could not assume that the overlapping fragments would be identical, especially since additional shapes were applied to the slices. So I used correlation. The place where the statistic was highest indicated the location being searched for. Of course, I didn't win the event because I spent half of the night correcting the script, but I still had fun.

Signal Processing on a DSP

Generating signals using Q15 numbers and designing a low-pass filter

It was a fascinating project! Writing a program for a fixed-point DSP-type processor is quite a challenge. Float data types were not allowed; Operations in Q15 numbers. We had to be careful with overflows and minimize division operations by bit shifting. Working only on whole arrays; It was a quite new innovation for me. Before that project I was used to processing arrays only in a serial mode - by using loops. For sound processing such fashion would result in an increasing output delay, making real time impossible. The sampling rate of audio signals for DVD standard is 48 kHz. Therefore, the number of effective CPU clock ticks to be used is 48,000 times smaller. For a 1 Mhz clock it would be 1M/48k = 1/0.048 = 20.8(3). This leaves a maximum of 20 operations to process each sample. It is worth keeping in mind that division and multiplication are broken down by the compiler into simpler additions and subtractions, hence division and multiplication take much more operations (processor ticks) from our poor 20 ops pool. It was a good introduction for me to later work with python numpy arrays and feeding models in batches while studying artificial intelligence specialization. Ironically, I was surprised to be welcomed back into a sort of serial mode in data loaders, when the training data became too large to be cached.

github

Let's break down some of the key aspects:

Fixed-Point Arithmetic (Q15):

  • In fixed-point arithmetic, numbers are represented with a fixed number of bits for the integer and fractional parts.
  • Q15 format uses 16 bits, with 15 bits for the fractional part and 1 bit for the sign.
  • This format allows efficient integer operations without the overhead of floating-point arithmetic.
  • Overflows can occur if you're not careful with the range of values.

Handling Overflows:

  • To prevent overflows, you likely used saturation arithmetic or compressor-expander techniques.
  • Saturation ensures that values stay within a specified range (e.g., -32768 to 32767 for Q15).
  • Compressor is a non-linear element, that compress dynamic of the signal. It multiply high amplitudes by the fraction value from corresponding multiplayer-amplitude compressor characteristic. It ensures signal never reach higher then limit value.
  • Compressor also can do the same thing with low values, but by gaining them. It ensures signal never drop under the low limit value.
  • Expander does the opposite operations.

Minimizing Division Operations:

  • Division is expensive in terms of computation time.
  • Bit shifting (right shift) can be used to approximate division by powers of 2.
  • For example, dividing by 2^n is equivalent to right-shifting n bits.

Signal Processing:

  • Working with sample arrays allowed you to process signals efficiently.
  • I've made different types od signals (sine waves, square waves, sawtooth wave)
  • I've designed and implemented low-pass filter.
  • We were also to work on real signals from microphone.
  • The low-pass frequency filter is essential for removing high-frequency noise.

Frequency field analysis

  • Using FFT (Fast Fourier Transform).
  • Balancing between time and freuancy resolutions.
  • Bigger analysis window - results higher frequency resolution (better) BUT lower time resolution (worse).
  • Smaller analysis window - results lower frequency resolution (worse) BUT higher time resolution (better).
  • Nyquist-Shannon sampling theorem relates the two parameters. To accurately reconstruct a signal, the sampling rate should be at least twice the bandwidth of the signal.
  • The maximum sound pass-bandwidth while 48 KHz sampling rate is 24 kHz.
Overall, this project demonstrates a deep understanding of DSP principles and optimization techniques. üòÅüé∂

Sawtooth signal generation

Amplitude step value calculations
step = (2*32768)/(48000/f) ‚âà (f*22368)/16384 = (f*22368)>>14
step = STEP_SAW = round((2*32768)/(48000/f)) = round(136,5333) = 137

Saw function code
                  
                    void saw(int* buffer, unsigned int buflen, int step)
                    {
                      static int accumulator = 0;
                      int i;
                      for(i = 0; i < buflen; i++)
                      {
                        buffer[i] = accumulator;
                        accumulator += step;
                      }
                    }

The actual frequency of the generated signal
(2*32768*f)/(48000) = 137
f = (137*48000)/(2*32768) = 100,3417969 Hz

The actual frequency of the generated signal is 100.3417969 Hz. The difference results from the precision of numerical notation in the computer. Greater precision can be achieved by sacrificing more memory to recording the number. We cannot obtain a frequency of exactly 100 Hz, because the number 136.5(3) does not have a finite decimal notation.
An inverted saw could be generated by subtracting the step instead of adding it to the accumulator in the saw function.

Generating a square wave signal

Rect function code
                  
                    void rect(int* buffer, unsigned int buflen, int step, int threshold)
                    {
                      saw(buffer, buflen, step);
                      int i;
                      for(i = 0; i < buflen; i++)
                      {
                        if (buffer[i] < threshold)
                          buffer[i] = -32768;
                        else
                          buffer[i] = 32767;
                      }
                    }

Generating a triangle signal

Tri function code
                  
                    void tri(int* buffer, unsigned int buflen, int step)
                    {
                      saw(buffer, buflen, step);
                      int i;
                      int temp;
                      for(i = 0; i < buflen; i++)
                      {
                        temp = buffer[i];
                        temp = temp < 0 ? -temp : temp;
                        buffer[i] = (temp - 16384)<<1;
                      }
                    }
